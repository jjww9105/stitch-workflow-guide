Semantic rebuild workflow

기존 UI 구조는 버리고, 기존 구현의 기능 계약은 살린다.

이 가이드는 Codex 프로젝트의 제품 의미와 실제 구현 계약을 추출하고, Stitch가 제안한 새 화면 구조를 기능이 깨지지 않는 방식으로 실제 프로젝트에 통합하기 위한 실행 보드다.

Step 01부터 시작 치트시트 열기
버릴 것 기존 프론트엔드 구조

레이아웃, 섹션 순서, 탭 배치, 정보 위계, 시각 스타일은 보존 대상이 아니다.

살릴 것 기존 구현의 기능 계약

API, 라우팅, 상태관리, 인증, validation, 에러 처리, 실제 데이터 흐름은 보존한다.

Decision rule

충돌이 생기면 이렇게 판단한다

  1. 01

    기존 UI 구조 vs Stitch 구조
    Stitch의 새 구조를 우선 검토한다.

  2. 02

    Stitch 구조 vs 기능 계약
    기능 계약이 이긴다.

  3. 03

    기능을 살려야 할 때
    기존 UI로 되돌아가지 않고 새 구조 안에서 다시 배치한다.

Execution board

6단계 실행 보드

각 단계는 붙이는 위치, 입력물, 결과물, 완료 기준을 함께 본다.

결과

Stitch

화면 커버리지와 기능 계약 검사하기
