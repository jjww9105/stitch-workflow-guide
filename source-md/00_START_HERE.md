# Start Here

이 문서는 Google Stitch와 Codex를 함께 써서 기존 프로젝트의 새 프론트엔드를 만드는 워크플로우다.

가장 중요한 원칙은 하나다.

```text
기존 UI 구조는 믿지 않는다.
기존 구현의 기능 계약은 믿는다.
```

## 이 워크플로우가 해결하려는 문제

기존 Codex 프로젝트에는 이미 기능이 구현되어 있을 수 있다.  
하지만 Codex가 만든 프론트엔드가 보기 어렵거나, 정보구조가 이상하거나, 화면 배치가 구릴 수 있다.

이때 단순히 "예쁘게 리디자인해줘"라고 하면 기존 UI의 나쁜 구조가 남는다.

반대로 기존 UI를 전부 무시하면 실제 기능, API, 상태관리, 권한, 검증, 에러 처리가 깨질 수 있다.

그래서 이 워크플로우는 두 가지를 분리한다.

## 버릴 것과 보존할 것

| 구분 | 처리 |
|---|---|
| 기존 UI 레이아웃 | 버림 |
| 기존 섹션 순서 | 버림 |
| 기존 탭 배치 | 버림 |
| 기존 정보 위계 | 버림 |
| 기존 색상/폰트/컴포넌트 모양 | 버림 |
| 기존 프롬프트/기능 나열 방식 | 버림 |
| 제품 목적 | 보존 |
| 사용자 흐름 | 보존 |
| 화면/라우트/상태의 존재 | 보존 |
| API 연동 | 보존 |
| 상태관리 | 보존 |
| 인증/권한 | 보존 |

## 전체 흐름

```text
1. Codex가 제품 의미와 기능 계약을 추출한다
2. Stitch가 기존 UI 구조를 버리고 새 전체 화면 구조를 만든다
3. Stitch가 레퍼런스 이미지에서 DESIGN.md를 만든다
4. Stitch가 새 전체 화면 세트에 DESIGN.md를 적용한다
5. Stitch에서 화면 커버리지와 기능 계약 보존을 검사한다
6. zip을 다운로드한다
7. Codex가 Stitch의 새 구조를 실제 프로젝트에 통합한다
8. Codex가 기능 계약이 깨지지 않았는지 빌드/테스트/화면으로 검증한다
```

## 각 단계의 역할

| Step | 붙이는 곳 | 역할 | 결과물 |
|---|---|---|---|
| 01 | Codex | 제품 의미 + 기능 계약 추출 | Semantic Rebuild Brief + Functional Contract |
| 02 | Stitch | 새 전체 화면 구조 생성 | 전체 화면 세트 + 기능 보존 매핑표 |
| 03 | Stitch | 레퍼런스 이미지에서 디자인 시스템 추출 | DESIGN.md |
| 04 | Stitch | 전체 화면 세트에 DESIGN.md 적용 | 디자인 적용된 전체 화면 세트 |
| 05 | Stitch | 커버리지/기능/상태/반응형 검사 | 누락 목록 + 수정된 화면 |
| 06 | Codex | Stitch zip을 실제 프로젝트에 통합 | 작동하는 새 프론트엔드 |

## Step 01에서 꼭 나와야 하는 것

`Semantic Rebuild Brief`만 있으면 부족하다.

복잡한 프로젝트에서는 반드시 `Functional Contract`가 있어야 한다.

Functional Contract는 이런 내용이다.

```text
이 버튼은 어떤 API를 호출하는가?
이 화면은 어떤 데이터를 읽는가?
이 폼은 어떤 validation을 갖는가?
권한이 없으면 무엇을 보여주는가?
성공/실패/로딩/빈 상태는 어떻게 처리하는가?
이 기능이 깨졌다고 판단하는 기준은 무엇인가?
```

## Stitch가 할 일과 못 하는 일

Stitch가 잘하는 일:

- 새 화면 구조 제안
- 새 배치와 정보구조 제안
- 전체 화면 세트 생성
- 디자인 시스템 적용
- 여러 상태 화면을 시각적으로 구성

Stitch가 완벽히 알 수 없는 일:

- 실제 API의 세부 동작
- 상태관리 내부 규칙
- 인증/권한 처리
- 기존 코드의 숨은 validation
- 실제 빌드/런타임 제약

그래서 마지막 통합은 Codex가 해야 한다.

## 충돌이 생겼을 때 기준

```text
Stitch 구조가 기존 UI 구조와 충돌한다
-> Stitch 구조를 우선 검토한다.

Stitch 구조가 기능 계약과 충돌한다
-> 기능 계약이 이긴다.

기능 계약을 지켜야 한다
-> 기존 UI로 되돌아가지 말고, 새 구조 안에서 기능을 다시 배치한다.
```

## 가장 흔한 실수

```text
기존 화면 구조를 그대로 두고 색만 바꾸는 것
```

이건 실패다.

```text
Stitch 화면을 그대로 덮어써서 기능이 깨지는 것
```

이것도 실패다.

성공 기준은 이거다.

```text
새 화면 구조가 생겼고,
기존 기능 계약도 그대로 작동한다.
```
