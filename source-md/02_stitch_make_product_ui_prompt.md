# 02. Stitch에서 전체 화면 세트를 만들 때 쓰는 프롬프트

사용 위치: Google Stitch

사용법:

1. Codex가 만든 `Semantic Rebuild Brief` 전체를 먼저 붙여넣는다.
2. 구분선 `---`를 넣는다.
3. 아래 프롬프트를 이어서 붙여넣는다.

```text
위에 붙인 Semantic Rebuild Brief를 바탕으로 이 제품의 새로운 프론트엔드 전체 화면 세트를 만들어줘.

언어 규칙:
- 모든 화면 문구, 메뉴, 버튼, 라벨, 안내문, 상태 메시지, 예시 데이터, 설명은 한국어로 작성해.
- 필요한 고유명사와 도구명은 영어를 유지해도 돼: Codex, Stitch, DESIGN.md, GitHub Pages, zip, Copy.
- 최종 사용자는 한국어로 이 제품을 사용하는 사람이라고 가정해.

중요 규칙:
기존 Codex UI는 시각 참조가 아니다. 기존 UI의 레이아웃, 색상, 타이포그래피, 컴포넌트 구조, 탭 배치 방식, 여백, 섹션 순서, 정보 위계, 장식 스타일을 보존하거나 개선하거나 따라 하지 마.

기존 UI에서 사용할 수 있는 것은 오직 제품 의미와 화면 커버리지뿐이야:
- 제품 목적
- 기능 목록
- 탭/라우트/화면의 존재
- 사용자 흐름
- 화면 의도
- 데이터 상태
- 누락 상태
- 피해야 할 anti-pattern

목표:
제품 의미와 사용자 흐름에 맞는 완전히 새로운 전체 앱 구조와 전체 화면 세트를 만들어줘.
결과물은 Codex가 실제 구현에 참고할 수 있는 화면 설계 재료여야 한다.

먼저 할 일:
1. Brief에서 전체 탭/라우트/화면/상태 인벤토리를 추출해.
2. Stitch가 만들어야 할 화면 목록을 `화면 커버리지 표`로 정리해.
3. 각 화면을 필수 화면, 보조 화면, 상태 화면으로 분류해.
4. 누락하면 실제 개발이 막히는 화면을 표시해.

생성할 것:
1. 전체 앱 구조
   - 전역 네비게이션
   - 탭/라우트 구성
   - 주요 사용자 흐름
   - 화면 간 이동 방식

2. 전체 화면 세트
   - Brief에 나온 모든 핵심 탭
   - 모든 핵심 라우트/페이지
   - 필요한 상세 화면
   - 필요한 모달/드로어/설정/온보딩
   - 각 화면의 핵심 콘텐츠와 주요 액션

3. 상태 화면
   - loading
   - empty
   - error
   - success / confirmation
   - permission / locked
   - validation

4. 모바일/데스크톱 방향
   - 전체 앱 구조가 작은 화면과 큰 화면에서 어떻게 바뀌는지
   - 탭/라우트/테이블/폼이 어떻게 적응하는지

5. 현실적인 한국어 예시 콘텐츠
   - 화면별 제목
   - 버튼/라벨/상태 메시지
   - 예시 데이터

배치 생성 규칙:
- 한 번에 모든 화면을 만들 수 있으면 전체 화면 세트를 한 번에 만들어.
- 한 번에 다 만들 수 없으면 대표 화면만 고르고 끝내지 마.
- 대신 Batch 1부터 만들고, 마지막에 반드시 다음 표를 남겨:
  1. 이번 배치에서 만든 화면
  2. 아직 남은 화면
  3. 다음 배치에서 만들어야 할 화면
  4. 아직 생성되지 않은 상태 화면

피할 것:
- 일부 화면만 예쁘게 만들고 전체 제품을 덮었다고 말하기
- Brief가 요구한 탭/라우트/핵심 기능 누락
- 사용자 작업에 도움이 안 되는 장식 섹션
- 기존 Codex UI의 시각적 흔적
- Brief에 없는 가짜 기능
- 영어 UI

출력:
1. 화면 커버리지 표
2. 새 전체 앱 구조
3. 생성한 전체 화면 세트
4. 상태 화면 목록
5. 만든 화면 / 남은 화면 추적표
6. Codex 구현에 필요한 가정
```
