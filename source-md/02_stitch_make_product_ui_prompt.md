# 02. Stitch에서 새 전체 화면 구조 만들기

사용 위치: Google Stitch

목적:

Codex가 만든 `Semantic Rebuild Brief + Functional Contract`를 바탕으로 기존 UI 구조를 따라 하지 않는 새 전체 화면 세트를 만든다.

사용법:

1. Codex가 만든 `Semantic Rebuild Brief + Functional Contract` 전체를 먼저 붙여넣는다.
2. 구분선 `---`를 넣는다.
3. 아래 프롬프트를 이어서 붙여넣는다.

```text
위에 붙인 `Semantic Rebuild Brief + Functional Contract`를 바탕으로 이 제품의 새로운 프론트엔드 전체 화면 구조와 전체 화면 세트를 만들어줘.

언어 규칙:
- 모든 화면 문구, 메뉴, 버튼, 라벨, 안내문, 상태 메시지, 예시 데이터, 설명은 한국어로 작성해.
- 필요한 고유명사와 도구명은 영어를 유지해도 돼: Codex, Stitch, DESIGN.md, GitHub Pages, zip, Copy.
- 최종 사용자는 한국어로 이 제품을 사용하는 사람이라고 가정해.
- 영어 UI로 생성하지 마.

핵심 원칙:
기존 Codex UI는 시각 참조도, 정보구조 참조도 아니다.
기존 UI의 레이아웃, 색상, 타이포그래피, 컴포넌트 구조, 탭 배치 방식, 여백, 섹션 순서, 정보 위계, 장식 스타일을 보존하거나 개선하거나 따라 하지 마.

기존 UI에서 사용할 수 있는 것은 오직 제품 의미와 기능적 증거뿐이야:
- 제품 목적
- 기능 목록
- 탭/라우트/화면의 존재
- 사용자 흐름
- 화면 의도
- 데이터 상태
- 누락 상태
- 피해야 할 anti-pattern
- Functional Contract에 적힌 필수 동작

기능 계약 규칙:
- Functional Contract에 있는 기능, 주요 액션, 데이터 흐름, validation, permission, loading/empty/error/success 상태를 삭제하지 마.
- 기능을 합치거나 이름을 바꾸거나 다른 화면으로 옮길 수는 있지만, 반드시 `기존 기능 -> 새 화면 위치` 매핑을 남겨.
- 실제 구현과 충돌할 수 있는 제안은 `구현 확인 필요`로 표시해.
- 기능을 임의로 단순화하지 마.
- 더미 기능이나 Brief에 없는 가짜 기능을 만들지 마.

목표:
제품 의미, 전체 화면 커버리지, 기능 계약을 바탕으로 완전히 새로운 앱 구조와 전체 화면 세트를 만들어줘.
결과물은 Codex가 실제 구현에 통합할 수 있는 화면 구조/배치/흐름 제안이어야 한다.

먼저 할 일:
1. Brief에서 전체 탭/라우트/화면/상태 인벤토리를 추출해.
2. Functional Contract에서 절대 삭제하면 안 되는 기능과 액션을 추출해.
3. 기존 UI 구조를 버리고, 사용자 작업 기준의 새 정보구조를 제안해.
4. 화면 목록을 `화면 커버리지 표`로 정리해.
5. 기능을 새 화면에 어떻게 배치했는지 `기능 보존 매핑표`로 정리해.

생성할 것:

1. 새 전체 앱 구조
   - 전역 네비게이션
   - 탭/라우트 구성
   - 주요 사용자 흐름
   - 화면 간 이동 방식
   - 기존 UI 구조와 달라진 점

2. 전체 화면 세트
   - Brief에 나온 모든 핵심 탭
   - 모든 핵심 라우트/페이지
   - 필요한 상세 화면
   - 필요한 모달/드로어/설정/온보딩
   - 각 화면의 핵심 콘텐츠와 주요 액션

3. 기능 보존 매핑
   - Functional Contract의 각 기능
   - 새 UI에서 배치된 화면
   - 사용자가 누르는 액션
   - 필요한 데이터/상태
   - 구현 확인이 필요한 점

4. 상태 화면
   - loading
   - empty
   - error
   - success / confirmation
   - permission / locked
   - validation

5. 모바일/데스크톱 방향
   - 전체 앱 구조가 작은 화면과 큰 화면에서 어떻게 바뀌는지
   - 탭/라우트/테이블/폼/긴 프롬프트가 어떻게 적응하는지

6. 현실적인 한국어 예시 콘텐츠
   - 화면별 제목
   - 버튼/라벨/상태 메시지
   - 예시 데이터

배치 생성 규칙:
- 한 번에 모든 화면을 만들 수 있으면 전체 화면 세트를 한 번에 만들어.
- 한 번에 다 만들 수 없으면 대표 화면만 고르고 끝내지 마.
- 대신 Batch 1부터 만들고, 마지막에 반드시 다음 표를 남겨:
  1. 이번 배치에서 만든 화면
  2. 이번 배치에서 보존한 기능 계약
  3. 아직 남은 화면
  4. 아직 배치되지 않은 기능 계약
  5. 다음 배치에서 만들어야 할 화면
  6. 아직 생성되지 않은 상태 화면

피할 것:
- 일부 화면만 예쁘게 만들고 전체 제품을 덮었다고 말하기
- Brief가 요구한 탭/라우트/핵심 기능 누락
- Functional Contract의 기능 삭제, 임의 병합, 숨김 처리
- 사용자 작업에 도움이 안 되는 장식 섹션
- 기존 Codex UI의 시각/구조 흔적
- Brief에 없는 가짜 기능
- 영어 UI

출력:
1. 화면 커버리지 표
2. 기능 보존 매핑표
3. 새 전체 앱 구조
4. 생성한 전체 화면 세트
5. 상태 화면 목록
6. 만든 화면 / 남은 화면 / 남은 기능 계약 추적표
7. Codex 구현에 필요한 가정과 충돌 가능성
```
