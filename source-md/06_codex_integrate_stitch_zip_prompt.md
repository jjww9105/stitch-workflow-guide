# 06. Stitch zip을 Codex 프로젝트에 통합하기

사용 위치: 실제 제품의 Codex 프로젝트 폴더

목적:

Stitch zip의 새 화면 구조/배치/흐름을 실제 프로젝트에 통합하되, 기존 구현의 기능 계약은 깨지지 않게 한다.

사용법:

1. Stitch에서 결과물을 zip으로 다운로드한다.
2. 실제 제품 프로젝트를 Codex에서 연다.
3. zip 파일 경로와 함께 아래 프롬프트를 붙여넣는다.

```text
Stitch에서 다운로드한 zip을 현재 프로젝트의 새 프론트엔드 구현 재료로 사용해서 통합해줘.

핵심 원칙:
- Stitch zip을 그대로 덮어쓰지 마.
- 기존 프로젝트의 기능, 라우팅, API 연동, 상태관리, 인증/권한, 데이터 처리, validation, 에러 처리는 보존해.
- 기존 프론트엔드의 레이아웃, 섹션 순서, 탭 배치, 정보구조, 컴포넌트 모양, 시각 스타일은 보존 대상이 아니다.
- Stitch zip은 단순 스타일 참고가 아니라 새 화면 구조/배치/흐름 제안으로 분석해.
- 기존 UI 구조와 Stitch 구조가 충돌하면 기존 UI 구조를 기본값으로 삼지 마.
- 단, Stitch 구조가 기능 계약을 깨뜨리면 기능 계약이 이긴다. 이때 기존 UI로 되돌아가지 말고, Stitch의 새 구조 안에서 기능이 작동하도록 재배치해.

먼저 할 일:
1. 현재 프로젝트의 실제 스택, 라우팅, 상태관리, 데이터 흐름, 빌드 설정을 확인해.
2. 현재 구현에서 Functional Contract를 다시 추출해.
3. 현재 프론트엔드에서 보존하면 안 되는 UI 구조/정보구조/시각 요소를 분리해.
4. Stitch zip 내부 구조, 화면 목록, 컴포넌트, 스타일, 데이터 더미를 확인해.
5. Semantic Rebuild Brief + Functional Contract가 있다면 함께 읽고 기준으로 삼아.

비교할 것:
1. 기존 구현의 기능 계약
2. 기존 UI에서 버려야 할 구조
3. Brief의 전체 화면/탭/라우트 커버리지
4. Stitch zip의 새 화면 구조/배치/흐름
5. Stitch zip의 누락 화면/누락 상태/누락 액션
6. 기능 계약과 Stitch 구조가 충돌하는 지점

통합 계획:
1. Stitch 구조에서 실제로 채택할 새 정보구조를 정리해.
2. 기존 기능 계약을 새 구조 안에 어떻게 배치할지 매핑해.
3. 어떤 파일/컴포넌트/스타일을 새로 만들거나 바꿀지 계획해.
4. 하드코딩된 더미 데이터가 있으면 실제 데이터 흐름에 맞게 바꾸는 방법을 정리해.
5. 기능 손상 위험이 있는 부분은 구현 전에 표시해.

구현 규칙:
- 기존 프로젝트 기능을 삭제하지 마.
- 사용자 변경사항을 되돌리지 마.
- Stitch 코드를 무비판적으로 복붙하지 마.
- Stitch zip의 일부 화면만 보고 전체 앱 통합이 끝났다고 판단하지 마.
- 기존 UI의 낡은 화면 구조로 되돌리지 마.
- 기능 계약을 지키기 위해 필요한 액션은 새 UI 안에서 다시 배치해.
- 레퍼런스 사이트의 브랜드/문구/로고를 복사하지 마.
- loading / empty / error / success / permission / validation 상태를 실제 코드에 반영해.
- 접근성, 반응형, 텍스트 overflow를 확인해.
- 빌드 또는 테스트를 실행해서 깨진 부분을 수정해.

충돌 처리 규칙:
- Stitch가 기능을 삭제했으면 기능을 복원하되 기존 UI 구조로 복원하지 말고 새 구조에 맞게 배치해.
- Stitch가 여러 기능을 하나로 합쳤는데 실제 구현상 분리되어야 하면 primary/secondary action, menu, drawer, detail view 등으로 다시 분리해.
- Stitch가 더미 데이터를 만들었으면 실제 API/state/data source에 연결해.
- Stitch가 실제 권한/validation/error 흐름을 모르면 기존 구현 계약을 기준으로 보강해.
- Stitch의 화면 구조가 좋지만 현재 구현 라우팅과 다르면, 라우팅을 안전하게 조정하거나 기존 라우트 위에 새 화면 구조를 매핑해.

출력:
1. 전체 화면/탭/라우트 커버리지 비교
2. Functional Contract 보존 매핑표
3. 기존 UI에서 버린 구조/시각 요소
4. Stitch zip에서 채택한 새 화면 구조/배치/흐름
5. Stitch zip에서 채택하지 않은 부분과 이유
6. 통합 계획
7. 적용한 파일 목록
8. 유지한 기존 기능
9. 수정한 UI/컴포넌트
10. 추가하거나 보강한 누락 화면/상태
11. 실행한 검증 명령
12. 남은 리스크

Stitch zip 경로:
[여기에 zip 경로 입력]
```
