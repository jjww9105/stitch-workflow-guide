# Semantic Rebuild Brief

## 0. Coverage Inventory

### 전체 화면/섹션 인벤토리

| 화면/섹션 | 유형 | 필수 여부 | 목적 | 새 UI에서 필요한 처리 |
|---|---|---|---|---|
| 시작/현재 작업 화면 | route/section | 필수 | 사용자가 이 사이트의 목적과 첫 행동을 즉시 이해 | 마케팅 히어로가 아니라 실행 시작 지점으로 구성 |
| 전체 워크플로우 지도 | section | 필수 | 제품 의미 라인과 디자인 감성 라인을 분리해서 보여줌 | 전체 앱 재생성 흐름과 배치 생성 흐름을 함께 보여줌 |
| Step 01 Brief 생성 | step screen | 필수 | Codex에서 제품 의미와 화면 인벤토리 생성 | Copy, 사용 위치, 입력물, 기대 결과 표시 |
| Step 02 전체 화면 세트 생성 | step screen | 필수 | Stitch에서 전체 앱 구조와 모든 핵심 화면 생성 | 화면 커버리지 표, 만든 화면/남은 화면 추적 강조 |
| Step 03 DESIGN.md 생성 | step screen | 필수 | 레퍼런스 이미지에서 전체 화면용 디자인 시스템 생성 | 원본 사이트 과복사 방지 규칙 표시 |
| Step 04 DESIGN.md 전체 적용 | step screen | 필수 | DESIGN.md를 전체 화면 세트에 적용 | 적용 커버리지 표와 남은 화면 표시 |
| Step 05 품질 검사 | step screen | 필수 | 커버리지, 오염, 과복사, 상태, 반응형 점검 | 여러 검사 프롬프트를 목적별 도구로 제공 |
| Step 06 zip 통합 | step screen | 필수 | Stitch zip을 실제 Codex 프로젝트에 통합 | 전체 화면/탭/라우트 기준으로 통합 안내 |
| 원본 md 자료 | support screen | 필수 | source-md 문서 접근 | 파일 역할과 링크 제공 |
| GitHub Pages 배포/백업 | support screen | 보조 | 사이트 수정 후 배포/백업 | 메인 흐름을 방해하지 않는 보조 정보로 제공 |
| Copy 성공/실패 상태 | state | 필수 | 복사 결과를 사용자에게 알려줌 | 한국어 토스트, 버튼 상태, 접근성 상태 제공 |
| 빈 상태/오류 상태 | state | 필수 | 프롬프트 누락, 원본 파일 링크 오류, 복사 실패 처리 | 복구 행동 안내 |

### 반드시 생성해야 하는 상태

- loading: 원본 문서나 프롬프트 로딩 중
- empty: 프롬프트 본문 또는 화면 목록이 비어 있음
- error: Copy 실패, 원본 md 링크 실패, 커버리지 누락
- success: Copy 완료, 단계 완료, 배포 확인
- permission: Clipboard 권한 제한
- validation: 제품 유형 placeholder나 zip 경로 누락

### 기존 UI에서 추출해도 되는 것

- 제품 목적
- 프롬프트 목록
- 단계 순서
- Codex용/Stitch용 도구 구분
- 원본 md 링크
- Copy 동작
- GitHub Pages 배포/백업 흐름
- 현재 UI의 문제와 anti-pattern

### 기존 UI에서 가져오면 안 되는 것

- 레이아웃
- 색상
- 폰트
- 카드 모양
- 컴포넌트 구조
- 탭/섹션 배치 방식
- 여백
- 정보 위계
- 장식 스타일

## 1. Product Snapshot

### 제품명

Stitch Workflow Guide

### 한 줄 목적

Codex 프로젝트의 제품 의미와 전체 화면 커버리지를 추출하고, Google Stitch에서 전체 화면 세트와 DESIGN.md를 결합해 실제 구현 가능한 새 프론트엔드 재료를 만들게 하는 한국어 실행 가이드 웹앱.

### 대상 사용자

- 주 사용자: Codex와 Google Stitch를 함께 사용해 기존 프로젝트의 프론트엔드를 새로 만들려는 개인 프로젝트 소유자.
- 보조 사용자: 몇 달 뒤 같은 절차를 다시 복구해야 하는 미래의 사용자 본인.
- 보조 사용자: 링크만 받고 절차를 이해해야 하는 Codex 에이전트나 협업자.

### 사용자가 해결하려는 문제

- Stitch가 일부 예쁜 화면만 만들고 전체 앱 구조를 빠뜨릴 수 있다.
- 어떤 프롬프트를 Codex에 붙이고 어떤 프롬프트를 Stitch에 붙여야 하는지 헷갈린다.
- 기존 Codex UI를 시각 참고자료로 착각하면 새 UI가 기존 UI에 오염된다.
- 레퍼런스 이미지를 잘못 쓰면 디자인 시스템이 아니라 원본 사이트 복제가 된다.
- Stitch zip을 실제 프로젝트에 통합할 때 전체 탭/라우트/화면 커버리지가 깨질 수 있다.

### 제품이 반드시 도와야 하는 일

- 전체 화면/탭/라우트 인벤토리를 만들게 한다.
- Stitch가 전체 화면 세트를 만들게 한다.
- 한 번에 다 못 만들면 배치로 나누고 만든 화면/남은 화면을 추적하게 한다.
- DESIGN.md가 전체 화면 세트에 적용되게 한다.
- 화면 커버리지 검사를 포함한다.
- Stitch zip 통합 시 일부 화면만 가져오는 실수를 막는다.
- 모든 사용자-facing UI를 한국어로 유지한다.

### 언어 규칙

- 최종 UI는 한국어다.
- 모든 화면 문구, 메뉴, 버튼, 라벨, 안내문, 상태 메시지, 예시 데이터는 한국어로 작성한다.
- Codex, Stitch, DESIGN.md, GitHub Pages, zip, Copy 같은 고유명사는 영어를 유지해도 된다.
- 영어 UI로 생성하지 않는다.

## 3. Core Workflows

### Workflow A. 전체 앱 재생성 브릿지

- 시작 지점: 사용자가 GitHub Pages 사이트를 연다.
- 단계:
  1. 전체 워크플로우와 절대 규칙을 확인한다.
  2. 실제 제품 프로젝트를 Codex에서 연다.
  3. Step 01로 Semantic Rebuild Brief와 전체 화면 인벤토리를 만든다.
  4. Stitch에 Brief와 Step 02를 넣어 전체 화면 세트를 만든다.
  5. 한 번에 다 못 만들면 만든 화면과 남은 화면을 추적하며 다음 배치로 이어간다.
  6. 레퍼런스 이미지를 올리고 Step 03으로 DESIGN.md를 만든다.
  7. Step 04로 전체 화면 세트에 DESIGN.md를 적용한다.
  8. Step 05로 화면 커버리지와 품질을 검사한다.
  9. zip을 다운로드하고 Step 06으로 실제 프로젝트에 통합한다.
- 성공 상태: 전체 화면/탭/라우트/상태가 빠짐없이 새 UI 구현 재료로 준비된다.
- 에러/막힘 상태:
  - 일부 화면만 만들어짐
  - 영어 UI로 생성됨
  - 기존 Codex UI 시각 흔적이 남음
  - 레퍼런스 사이트를 과복사함
  - zip 통합 시 기존 기능이 깨질 위험이 있음
- 복구 경로:
  - 화면 커버리지 검사 실행
  - 다음 배치 생성 지시
  - 기존 UI 오염 검사
  - 레퍼런스 과복사 검사
  - Codex 통합 전 커버리지 비교

### Workflow B. 프롬프트 복사

- 시작 지점: 사용자가 특정 Step을 연다.
- 단계:
  1. 사용 위치를 확인한다.
  2. 필요한 입력물을 확인한다.
  3. 프롬프트를 복사한다.
  4. Codex 또는 Stitch에 붙여넣는다.
  5. 기대 결과를 확인한다.
- 성공 상태: 잘린 프롬프트 없이 전체 본문이 복사된다.
- 에러/막힘 상태:
  - Copy 실패
  - 프롬프트 본문 누락
  - 잘못된 도구에 붙여넣음
- 복구 경로:
  - 직접 선택 복사 안내
  - 원본 md 열기
  - Codex/Stitch 배지 강화

## 6. Interaction Requirements

- 한 페이지에서 전체 흐름을 볼 수 있어야 한다.
- 현재 단계와 다음 행동이 명확해야 한다.
- Codex용, Stitch용, GitHub Pages용, 제품 프로젝트용 작업이 구분되어야 한다.
- 긴 프롬프트는 요약, 사용법, 본문, Copy를 분리해서 보여준다.
- 화면 커버리지 검사는 Step 05의 첫 검사로 배치한다.
- 모바일에서도 프롬프트와 Copy 버튼이 겹치지 않아야 한다.
- Copy 성공/실패 메시지는 한국어로 제공한다.
- 프롬프트 본문은 source-md와 불일치하지 않아야 한다.

## 7. Anti-Patterns From Current UI

| 문제 | 왜 문제인지 | 새 UI 해결 |
|---|---|---|
| 프롬프트가 일부만 들어갈 수 있음 | 사용자가 불완전한 지시를 복사할 수 있음 | source-md와 화면 본문 일치 검증 |
| 제품 UI 일부만 만드는 흐름 | 실제 프로젝트의 탭/라우트가 빠짐 | 전체 화면 인벤토리와 커버리지 검사 강조 |
| DESIGN.md를 한 화면에만 적용하는 흐름 | 전체 앱 일관성이 깨짐 | 전체 화면 세트 적용과 적용 커버리지 표 제공 |
| 기존 Codex UI를 따라갈 위험 | 새 UI가 구린 기존 UI에 오염됨 | 기존 UI는 의미/커버리지 추출용으로만 제한 |
| 레퍼런스 과복사 위험 | 원본 사이트 표절처럼 보일 수 있음 | 디자인 시스템 추출과 복사 금지 항목 분리 |
| 영어 메뉴/상태 혼입 | Stitch가 영어 UI를 만들 수 있음 | 한국어 UI 규칙을 모든 Stitch 프롬프트에 유지 |

## 8. Unknowns And Assumptions

### 확실하지 않은 점

- 최종 새 사이트가 단일 페이지 runbook인지 단계형 앱인지 아직 확정되지 않았다.
- 레퍼런스 디자인 방향은 사용자가 별도로 제공한다.
- Stitch export 구조는 생성 전까지 알 수 없다.

### 추정한 점

- 사용자는 이 사이트를 반복적으로 열어보는 실행 도구로 쓴다.
- 최종 UI는 한국어여야 한다.
- 공개 GitHub Pages 사이트이므로 민감한 프로젝트 정보는 넣지 않는다.
- Stitch 결과는 그대로 배포하지 않고 Codex가 실제 프로젝트에 통합한다.

### 사용자 확인이 필요한 점

- 진행률 저장이 필요한가?
- 프롬프트 편집 후 Copy 기능이 필요한가?
- 레퍼런스 디자인의 제품 유형은 무엇인가?
- GitHub Pages 배포 안내를 메인 흐름에 둘지 보조 문서로 둘지?

## Stitch에 처음 넣을 때 사용할 짧은 요약

GitHub Pages에 배포된 `Stitch Workflow Guide`를 위한 완전히 새로운 한국어 프론트엔드 콘셉트를 만들어줘.

기존 사이트 UI는 시각 참고자료가 아니다. 기존 레이아웃, 색상, 폰트, 카드, 여백, 장식, 정보 위계, 탭 배치 방식은 절대 보존하지 마. 기존 사이트에서는 제품 목적, 기능, 프롬프트 구조, 단계별 작업 의도, 전체 화면/탭/라우트 커버리지, 데이터 상태, 누락 상태, anti-pattern만 추출해.

이 제품은 Codex와 Google Stitch를 함께 써서 새 프론트엔드를 만드는 과정을 안내하는 실행형 runbook / prompt tool이다. 사용자는 사이트를 열고 Step 01부터 Step 06까지 따라가며 Semantic Rebuild Brief 생성, 전체 화면 세트 생성, 레퍼런스 이미지 기반 DESIGN.md 생성, DESIGN.md 전체 적용, 화면 커버리지 검사, Stitch zip의 Codex 통합을 진행한다.

최종 UI는 한국어여야 한다. 모든 화면 문구, 메뉴, 버튼, 라벨, 안내문, 상태 메시지, 예시 데이터는 한국어로 작성해. Codex, Stitch, DESIGN.md, GitHub Pages, zip, Copy 같은 고유명사는 영어를 유지해도 된다. 영어 UI로 만들지 마.

새 UI는 마케팅 랜딩페이지가 아니라, 사용자가 실제 작업 중 켜놓고 따라 할 수 있는 단계별 워크플로우 도구여야 한다. 각 단계는 사용 위치, 입력물, 복사할 프롬프트, 기대 결과, 다음 행동, 만든 화면/남은 화면, 막힘 상태와 복구 경로를 명확히 보여줘.
