# Semantic Rebuild Brief

## 1. Product Snapshot

### 제품명

Stitch Workflow Guide

### 한 줄 목적

Codex 프로젝트의 제품 의미와 레퍼런스 사이트의 디자인 감성을 분리해서 Google Stitch에 전달하고, 새 프론트엔드를 만들기 위한 프롬프트와 실행 순서를 한곳에서 따라 하게 해주는 한국어 워크플로우 가이드 웹앱.

### 대상 사용자

- 주 사용자: Codex와 Google Stitch를 함께 사용해서 기존 프로젝트의 프론트엔드를 새로 만들고 싶은 개인 프로젝트 소유자.
- 보조 사용자: 몇 달 뒤 같은 과정을 다시 해야 하는 미래의 사용자 본인.
- 보조 사용자: 사용자가 공유한 링크만 보고 절차를 이해해야 하는 Codex 에이전트나 협업자.

### 사용자가 해결하려는 문제

- Codex, Stitch, DESIGN.md, 레퍼런스 이미지, GitHub Pages, zip 통합까지 절차가 길어서 쉽게 잊힌다.
- 어떤 프롬프트를 Codex에 붙이고, 어떤 프롬프트를 Stitch에 붙여야 하는지 헷갈린다.
- 기존 Codex UI를 시각 참고자료로 착각하면, 새 UI가 기존 UI의 레이아웃과 스타일에 오염된다.
- 레퍼런스 사이트 캡처를 잘못 쓰면, 디자인 감성만 가져오는 것이 아니라 원본 사이트를 과하게 베끼게 된다.
- Stitch 결과물을 zip으로 받은 뒤 실제 프로젝트에 어떻게 통합해야 하는지 막힌다.
- 영어 프롬프트나 영어 UI 라벨이 섞이면, Stitch가 최종 결과물을 영어 UI로 생성할 수 있다.

### 제품이 반드시 도와야 하는 일

- 전체 절차를 처음부터 끝까지 한눈에 복구하게 한다.
- 각 단계의 사용 위치를 명확히 구분한다: Codex / Stitch / GitHub Pages / 실제 제품 프로젝트.
- 각 단계에서 필요한 입력물, 복사할 프롬프트, 기대 결과, 다음 행동을 보여준다.
- 프롬프트를 안정적으로 복사하게 한다.
- 레퍼런스 디자인 사용 시 복사 금지 항목과 추출 가능 항목을 분리해서 보여준다.
- 품질 검사 프롬프트의 용도와 사용 시점을 쉽게 설명한다.
- Stitch zip은 최종 코드가 아니라 실제 프로젝트에 통합할 재료라는 점을 반복해서 안내한다.
- 최종 Stitch 결과물의 사용자-facing UI 언어가 한국어가 되도록 강하게 지시한다.

### 언어 규칙

- 최종 Stitch 결과물의 사용자-facing UI 언어는 한국어다.
- 모든 화면 문구, 메뉴, 버튼, 라벨, 안내문, 상태 메시지, 예시 데이터는 한국어로 작성한다.
- Codex, Stitch, DESIGN.md, GitHub Pages, zip, Copy 같은 도구명/파일명/고유명사는 영어를 유지해도 된다.
- 영어 UI로 생성하지 않는다.
- 프롬프트 본문도 기본적으로 한국어로 보여준다.
- 영어 라벨이 필요한 경우에도 한국어 설명을 우선하고, 영어는 보조 표기로만 사용한다.

### 제품 성격

- 마케팅 랜딩페이지가 아니다.
- 문서 저장소만도 아니다.
- 사용자가 실제 작업 중 켜놓고 따라 하는 runbook / prompt tool / step-by-step workflow app이다.

## 3. Core Workflows

### Workflow A. 전체 Stitch 브릿지 실행

- 주요 사용자 흐름: 이 가이드 사이트를 보며 기존 프로젝트를 새 프론트엔드로 재구성한다.
- 시작 지점: 사용자가 GitHub Pages 사이트를 연다.
- 단계:
  1. 전체 흐름을 확인한다.
  2. 제품 프로젝트를 Codex에서 연다.
  3. Step 01 프롬프트를 복사해서 제품 프로젝트 Codex에 붙인다.
  4. Codex가 만든 Semantic Rebuild Brief를 복사한다.
  5. Stitch에 Brief 전체와 Step 02 프롬프트를 붙여 제품 UI 예시를 만든다.
  6. Stitch에 레퍼런스 사이트 캡처 이미지를 올린다.
  7. 이미지 선택 후 Step 03 프롬프트로 DESIGN.md/design system을 만든다.
  8. 제품 UI 예시와 DESIGN.md를 선택하고 Step 04 프롬프트로 UI를 재작업한다.
  9. 필요하면 Step 05 품질 검사 프롬프트를 하나씩 실행한다.
  10. Stitch zip을 다운로드한다.
  11. 제품 프로젝트 Codex에 zip과 Step 06 프롬프트를 전달해 통합한다.
- 성공 상태: 사용자가 새 UI 후보, 디자인 시스템, 검증된 Stitch 결과, Codex 통합 지시를 얻는다.
- 에러/막힘 상태:
  - 어떤 단계의 프롬프트를 써야 할지 모른다.
  - Stitch 결과물이 영어로 나온다.
  - Stitch가 기존 Codex UI를 따라 한다.
  - 레퍼런스 사이트를 너무 그대로 베낀다.
  - zip 통합 시 기존 기능이 깨질 위험이 있다.
- 복구 경로:
  - 현재 단계 표시와 "다음에 할 일"을 보여준다.
  - Codex용/Stitch용 배지를 명확히 보여준다.
  - 한국어 출력 규칙을 모든 Stitch 프롬프트에 포함한다.
  - 오염 검사, 과복사 검사, 상태 보강 프롬프트를 바로 실행할 수 있게 한다.
  - zip 통합 단계에서 "그대로 덮어쓰지 말 것"을 강하게 안내한다.

### Workflow B. Semantic Rebuild Brief 생성

- 주요 사용자 흐름: 현재 제품 프로젝트에서 Stitch가 이해할 제품 의미 문서를 만든다.
- 시작 지점: 사용자가 실제 제품 프로젝트 폴더를 Codex에서 연다.
- 단계:
  1. 사이트에서 Step 01을 연다.
  2. 프롬프트를 복사한다.
  3. 제품 프로젝트 Codex에 붙여넣는다.
  4. Codex가 프로젝트 파일, 화면, 코드, README 등을 읽고 제품 의미를 정리한다.
  5. 만들어진 Brief를 Stitch로 가져간다.
- 성공 상태: 기존 UI 스타일이 아니라 제품 목적, 기능, 사용자 흐름, 화면 의도, 데이터 상태가 정리된다.
- 에러/막힘 상태:
  - 사용자가 현재 사이트 자체와 대상 제품 프로젝트를 혼동한다.
  - Brief 안에 기존 UI 스타일 보존 지시가 섞인다.
  - 최종 UI 언어 규칙이 빠진다.
- 복구 경로:
  - Step 01에 "기존 UI는 시각 참조가 아니다"를 반복한다.
  - Brief 템플릿에 한국어 UI 규칙을 필수 항목으로 둔다.

### Workflow C. 레퍼런스 이미지에서 DESIGN.md 만들기

- 주요 사용자 흐름: 사용자가 원하는 디자인 감성의 사이트 캡처 이미지를 Stitch에서 디자인 시스템으로 바꾼다.
- 시작 지점: 사용자가 Stitch에 레퍼런스 이미지를 업로드한다.
- 단계:
  1. 이미지 선택.
  2. Step 03 프롬프트 붙여넣기.
  3. 색상, 타이포그래피, 여백, 컴포넌트, 상태 규칙 추출.
  4. 원본 레이아웃/브랜드/문구/로고를 복사하지 않았는지 확인.
- 성공 상태: 다른 제품 UI에 적용 가능한 독립적인 DESIGN.md/design system이 만들어진다.
- 에러/막힘 상태:
  - 사용자가 단순히 "design.md 생성해줘"라고만 입력한다.
  - Stitch가 원본 사이트의 섹션 구조와 브랜드 느낌을 너무 많이 복사한다.
- 복구 경로:
  - 추출 가능 항목과 복사 금지 항목을 프롬프트 안에 명시한다.
  - Step 05의 레퍼런스 과복사 검사를 제공한다.

### Workflow D. 제품 UI에 DESIGN.md 적용 및 품질 검사

- 주요 사용자 흐름: Stitch에서 만든 제품 UI 후보에 레퍼런스 기반 디자인 시스템을 입히고 품질을 점검한다.
- 시작 지점: Stitch에 제품 UI 후보와 DESIGN.md/design system이 모두 있다.
- 단계:
  1. 제품 UI 후보 선택.
  2. DESIGN.md/design system 선택.
  3. Step 04 프롬프트 붙여넣기.
  4. 필요한 Step 05 검사 실행: 기존 UI 오염, 레퍼런스 과복사, 상태 보강, DESIGN.md 준수, 반응형 검사.
- 성공 상태: 제품 의미는 유지되고, 시각 시스템은 DESIGN.md를 따르며, 한국어 UI 상태가 보강된다.
- 에러/막힘 상태:
  - 색상만 바뀌고 구조/컴포넌트 품질은 그대로다.
  - 상태 화면이 없다.
  - 모바일에서 텍스트나 컨트롤이 깨진다.
- 복구 경로:
  - 품질 검사 프롬프트를 별도 버튼/카드로 분리한다.
  - 각 검사 프롬프트의 목적과 사용 시점을 쉽게 설명한다.

### Workflow E. Stitch zip을 제품 프로젝트에 통합

- 주요 사용자 흐름: Stitch 결과를 실제 제품 코드에 반영한다.
- 시작 지점: 사용자가 Stitch에서 zip을 다운로드했다.
- 단계:
  1. 제품 프로젝트 Codex에 zip 경로를 준다.
  2. Step 06 프롬프트를 붙인다.
  3. Codex가 현재 프로젝트 구조와 Stitch zip 구조를 비교한다.
  4. 기능, 라우팅, 상태관리, 데이터 흐름을 보존하면서 UI 아이디어만 통합한다.
  5. 빌드/테스트/실행 확인을 한다.
- 성공 상태: 실제 프로젝트에 새 프론트엔드가 안정적으로 통합된다.
- 에러/막힘 상태:
  - zip을 그대로 덮어쓴다.
  - 더미 데이터가 실제 데이터 흐름을 덮어쓴다.
  - 기존 기능이 사라진다.
- 복구 경로:
  - Step 06에 "그대로 덮어쓰지 말 것"과 "기존 기능 보존"을 필수 규칙으로 둔다.

## 6. Interaction Requirements

### 네비게이션

- 사용자는 한 페이지에서 전체 흐름을 볼 수 있어야 한다.
- 동시에 "지금 내가 몇 단계에 있는지"가 보여야 한다.
- 데스크톱에서는 좌측 또는 상단에 단계 네비게이션을 제공한다.
- 모바일에서는 단계가 길어지므로 현재 단계 중심의 stepper 또는 compact navigation을 제공한다.
- Codex용, Stitch용, GitHub Pages용, 제품 프로젝트용 작업이 시각적으로 명확히 구분되어야 한다.
- 영어 메뉴만 쓰지 말고 한국어 메뉴를 우선한다.

### CTA

- 각 단계의 주 CTA는 하나여야 한다.
- 예: "Step 01 복사", "Stitch에 붙일 프롬프트 복사", "품질 검사 열기", "zip 통합 프롬프트 복사".
- Copy 버튼은 고유명사라 영어를 써도 되지만, 한국어 보조 문구를 함께 제공하는 편이 좋다.
- 복사 성공/실패 상태를 한국어로 보여준다.

### 폼/검증

- 복잡한 회원가입이나 저장 폼은 필요 없다.
- 필요한 경우만 짧은 입력 필드를 둔다.
- 예: 제품 유형, 레퍼런스 디자인 설명, zip 경로 메모, 현재 단계 메모.
- 필수 입력이 비어 있으면 한국어 validation 메시지를 보여준다.
- 프롬프트 안의 `[여기에 제품 유형 입력]` 같은 변수는 사용자가 놓치지 않게 강조한다.

### 검색/필터/정렬

- 프롬프트가 많으므로 필터가 유용하다.
- 필터 기준:
  - Codex용
  - Stitch용
  - 품질 검사
  - 배포/백업
  - zip 통합
- 검색 기준:
  - "DESIGN.md"
  - "한국어"
  - "오염 검사"
  - "레퍼런스"
  - "zip"
- 정렬은 기본적으로 실제 실행 순서를 따른다.

### 모달/드로어

- 긴 프롬프트는 화면을 압도하지 않게 상세 패널이나 drawer로 열 수 있다.
- 단, 사용자가 복사하기 전에는 전체 프롬프트 내용을 확인할 수 있어야 한다.
- 품질 검사 프롬프트는 목적 설명과 본문을 분리해서 보여준다.
- 원본 md 링크는 보조 정보로 제공한다.

### 반응형 요구사항

- 모바일에서도 Step 카드, 프롬프트, Copy 버튼이 겹치지 않아야 한다.
- 긴 프롬프트는 모바일에서 가로 스크롤이 아니라 세로 읽기가 가능해야 한다.
- 버튼 텍스트가 좁은 화면에서 잘리지 않아야 한다.
- 프롬프트 박스는 최소 높이와 확장 동작이 안정적이어야 한다.
- 텍스트가 많기 때문에 정보 밀도를 유지하되, 한 화면에 너무 많은 박스를 동시에 열어두지 않는다.

### 접근성/사용성

- Copy 버튼, stepper, details/drawer는 키보드로 접근 가능해야 한다.
- 복사 결과는 시각적 토스트뿐 아니라 screen reader가 읽을 수 있는 상태 영역에도 반영한다.
- 색상만으로 Codex/Stitch 단계를 구분하지 않는다.
- 긴 문서형 콘텐츠는 제목 계층이 명확해야 한다.

## 7. Anti-Patterns From Current UI

| 현재 UI에서 버려야 할 문제 | 왜 문제인지 | 새 UI에서는 어떻게 해결해야 하는지 |
|---|---|---|
| 기존 사이트를 랜딩페이지처럼 시작하는 구조 | 사용자는 설득을 원하는 것이 아니라 바로 작업을 시작하고 싶다. | 첫 화면에서 목적, 현재 할 일, 전체 단계, 첫 CTA를 실행형으로 보여준다. |
| 영어 메뉴와 영어 섹션명이 섞임 | Stitch가 영어 UI를 자연스러운 기본값으로 받아들일 수 있다. | 메뉴, 섹션명, 상태 메시지, 안내문은 한국어를 기본으로 쓴다. 고유명사만 영어 유지. |
| Codex/Stitch 단계 구분이 약함 | 사용자가 프롬프트를 잘못된 도구에 붙일 수 있다. | 각 단계에 사용 위치 배지, 입력물, 결과물, 다음 행동을 명확히 표시한다. |
| 프롬프트가 긴 textarea 안에만 들어 있음 | 읽기 어렵고, 모바일에서 관리하기 어렵고, 일부 본문이 누락되면 사용자가 알아차리기 어렵다. | 요약, 사용법, 본문, 변수, 복사 버튼을 분리한 프롬프트 상세 UI를 만든다. |
| 일부 프롬프트 본문이 현재 HTML에서 짧게 잘려 보일 수 있음 | 사용자가 Copy를 눌렀을 때 불완전한 프롬프트를 복사할 위험이 있다. | source-md를 source of truth로 삼고, 프롬프트 본문 누락/불일치 상태를 감지하거나 명확히 표시한다. |
| Step 05 품질 검사 프롬프트의 사용 시점이 직관적이지 않음 | 사용자가 모든 검사를 언제, 어디에, 몇 번 써야 하는지 헷갈린다. | 검사별 목적, 사용 타이밍, 기대 결과를 작은 도구 카드처럼 보여준다. |
| "DESIGN.md 만들기"와 "DESIGN.md 적용하기"가 초보자에게 비슷하게 보임 | 사용자가 디자인 시스템 생성 단계와 적용 단계를 혼동할 수 있다. | 레퍼런스 이미지 -> DESIGN.md 생성, 제품 UI + DESIGN.md -> 재작업을 두 개의 별도 흐름으로 시각화한다. |
| 현재 UI의 시각 스타일이 새 UI 방향에 영향을 줄 수 있음 | 이 프로젝트의 목적은 기존 UI 리디자인이 아니라 의미 기반 재생성이다. | 기존 UI의 레이아웃, 색상, 폰트, 카드, 여백, 정보 위계는 모두 폐기한다. |
| 단계별 성공/실패 상태가 부족함 | 사용자가 현재 단계가 완료되었는지, 다음에 무엇을 해야 하는지 알기 어렵다. | 각 단계에 성공 결과, 막힘 상태, 복구 행동을 표시한다. |
| 배포/백업 안내와 Stitch 실행 안내가 같은 밀도로 섞임 | 핵심 작업 흐름과 보조 관리 정보가 경쟁한다. | 메인 흐름은 Stitch 실행에 집중하고, GitHub Pages 배포/백업은 보조 섹션으로 분리한다. |
| Copy 성공 상태가 일시적인 토스트에만 의존함 | 사용자가 복사가 되었는지 놓칠 수 있다. | 버튼 상태, 토스트, 접근성 live region을 함께 사용한다. |
| 원본 md 파일의 역할이 한눈에 들어오지 않음 | 사용자가 `source-md` 파일을 전부 열어봐야 한다. | 파일명, 붙이는 곳, 하는 일, 사용 시점을 테이블이나 카드로 설명한다. |

## 8. Unknowns And Assumptions

### 확실하지 않은 점

- 사용자가 최종 사이트를 완전한 stepper 앱으로 원하는지, 아니면 한 페이지 runbook 형태를 선호하는지 확정되지 않았다.
- 사용자가 참고할 레퍼런스 사이트의 종류와 시각 방향은 아직 제공되지 않았다.
- Stitch가 생성할 구체적인 코드 스택과 export 구조는 현재 알 수 없다.
- 최종 결과를 GitHub Pages 정적 사이트로 계속 유지할지, 다른 프레임워크로 옮길지는 확정되지 않았다.
- 사용자가 진행 상태를 브라우저에 저장하기를 원하는지 알 수 없다.

### 추정한 점

- 사용자는 이 사이트를 반복적으로 열어보는 개인 실행 도구로 쓴다.
- 사용자는 한국어 UI를 원한다.
- 사용자는 기존 사이트의 디자인을 유지하고 싶지 않다.
- 사용자는 레퍼런스 디자인을 별도로 Stitch에 제공할 것이다.
- 사용자는 Stitch 결과를 그대로 배포하지 않고 Codex를 통해 실제 프로젝트 구조에 맞게 통합할 것이다.
- 사이트는 공개 GitHub Pages에 올라가므로 민감한 프로젝트 내용은 넣지 않는다.

### 사용자 확인이 필요한 점

- 새 UI가 "한 화면에서 전체를 펼쳐 보는 방식"이어야 하는가, 아니면 "단계별 마법사/체크리스트 방식"이어야 하는가?
- 진행률 저장, 완료 체크, 로컬 메모 기능이 필요한가?
- 프롬프트를 사이트 안에서 직접 편집해서 복사하는 기능이 필요한가?
- 레퍼런스 디자인 후보가 어떤 분위기인지: 업무용 SaaS, 개발자 도구, 문서형 앱, 에디터형 앱, 대시보드형 앱 중 무엇에 가까운가?
- GitHub Pages 배포 안내를 메인 기능으로 유지할지, 보조 문서로 낮출지 결정해야 한다.

## Stitch에 처음 넣을 때 사용할 짧은 요약

GitHub Pages에 배포된 `Stitch Workflow Guide`를 위한 완전히 새로운 한국어 프론트엔드 콘셉트를 만들어줘.

기존 사이트 UI는 시각 참고자료가 아니다. 기존 레이아웃, 색상, 폰트, 카드, 여백, 장식, 정보 위계는 절대 보존하지 마. 기존 사이트에서는 제품 목적, 기능, 사용자 흐름, 프롬프트 구조, 단계별 작업 의도, 데이터 상태, 누락 상태, anti-pattern만 추출해.

이 제품은 Codex와 Google Stitch를 함께 써서 새 프론트엔드를 만드는 과정을 안내하는 실행형 runbook / prompt tool이다. 사용자는 사이트를 열고 Step 01부터 Step 06까지 따라가며 Semantic Rebuild Brief 생성, Stitch UI 후보 생성, 레퍼런스 이미지 기반 DESIGN.md 생성, DESIGN.md 적용, 품질 검사, Stitch zip의 Codex 통합을 진행한다.

최종 UI는 한국어여야 한다. 모든 화면 문구, 메뉴, 버튼, 라벨, 안내문, 상태 메시지, 예시 데이터는 한국어로 작성해. Codex, Stitch, DESIGN.md, GitHub Pages, zip, Copy 같은 고유명사는 영어를 유지해도 된다. 영어 UI로 만들지 마.

새 UI는 마케팅 랜딩페이지가 아니라, 사용자가 실제 작업 중 켜놓고 따라 할 수 있는 단계별 워크플로우 도구여야 한다. 각 단계는 사용 위치, 입력물, 복사할 프롬프트, 기대 결과, 다음 행동, 막힘 상태와 복구 경로를 명확히 보여줘.
